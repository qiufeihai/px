# AI 开发规则与约束

本文件用于约束后续 AI 或自动化助手对本仓库的改动。

如果和别的默认习惯冲突，以本文件为准。

## 第一原则

任何改动都必须先满足这一条：

- 以最高极致性能方案为首
- 以低延迟、低抖动、高吞吐、高稳定为首
- 以最好能无损接近直连为首

如果一个方案会明显增加路径长度、状态复杂度、兼容层、调度开销或抖动风险，就不应该默认采用。

## 做决策时的顺序

遇到多个可选方案时，优先级按下面排序：

1. 性能与稳定性
2. 简单直接的数据路径
3. 可维护性
4. 易用性
5. 功能丰富度

不要为了“更通用”“更灵活”“更好扩展”而默认牺牲性能。

## 必须遵守的约束

- 客户端正式产品只有 GUI：`PX 个人代理`
- GUI 只做控制面，不应进入代理数据热路径
- 共享 runtime 才是客户端实际运行核心
- 首版只支持 TCP，不支持 UDP
- TUN 仅允许以外部 helper 方式接入，GUI 只负责启停和状态，不得把 TUN 协议栈塞进 runtime 热路径
- 不要引入大而全兼容层，不要把项目做成通用平台代理框架
- 不要为了兼容旧路径保留无意义分支
- 能删代码解决的问题，不要优先靠加参数和加开关解决

## 改代码前先判断

每次改动前，先回答这几个问题：

1. 会不会让数据路径变长？
2. 会不会增加额外线程、锁、队列、复制、序列化或上下文切换？
3. 会不会增加配置复杂度、分支复杂度或兼容负担？
4. 会不会让时延、抖动、吞吐或稳定性变差？

只要其中任何一项答案偏负面，就应该先换更简单的方案。

## 推荐的改动方向

- 优先减少层级和中间抽象
- 优先减少无效分支和兼容代码
- 优先减少动态策略和可变路径
- 优先把工作放在冷路径，而不是热路径
- 优先用明确约束代替模糊“灵活性”

## 不推荐的改动方向

- 为未来不确定需求预埋大块抽象
- 为少量边缘场景引入重型依赖
- 在热路径增加日志、统计、事件广播、消息转发
- 增加不必要的后台任务、守护线程或轮询
- 引入多协议、多入口、多控制面并行维护

## 文档与配置同步规则

- 新增或修改配置字段时，必须同步更新配置示例和相关文档
- 删除功能时，必须同步删除对应文档、脚本说明和失效分支
- 不要让文档停留在旧架构上

## 验证规则

完成改动后，至少做和改动范围匹配的验证：

- 代码改动后先跑最小必要检查
- 改过 Rust 代码时，至少做 `cargo check`
- 改过 Tauri 相关代码后，检查对应文件诊断
- 改文档时，保证描述与当前实现一致

## 沟通规则

- 默认给出最简单、最快、最稳定的方案
- 如果更高性能方案和更省事方案冲突，优先说明高性能方案
- 如果需求本身会破坏性能目标，要明确指出风险，不要默认执行

一句话总结：

这个项目不是为了做得更全，而是为了在个人使用场景里尽量更快、更稳、更接近直连。
